# 联调检查点（系统管理）

## 0. 联调前置
- 后端：`user-management-backend` 启动成功（默认 `http://localhost:8080`）
- 前端：`user-management-font` 启动成功（默认 `http://localhost:5173`，已配置 `/api` 代理到 8080）
- 数据库：PostgreSQL 数据库 `user-management` 可连接，表结构已初始化（后端启动会自动执行 `schema.sql` 的 `CREATE TABLE IF NOT EXISTS`）

## 1. 认证与会话（Captcha + JWT）

### 1.1 获取验证码
- 请求：`GET /api/auth/captcha`
- 检查点：
  - 返回 200，包含 `captchaId` 与 `code`
  - 验证码有时效（过期后登录应失败）

### 1.2 登录
- 请求：`POST /api/auth/login`
- Body：
  - `username`
  - `password`
  - `captchaId`
  - `captcha`（填 `code`）
- 检查点：
  - 返回 200，包含 `token`、`user`、`menus`、`permissions`
  - `token` 为 JWT，后续接口需要 `Authorization: Bearer <token>`
  - 登录失败会写入登录日志（后续在日志接口可查）

### 1.3 获取当前用户
- 请求：`GET /api/auth/me`
- Header：`Authorization: Bearer <token>`
- 检查点：
  - 返回 200，用户信息与登录返回一致

## 2. 组织管理（树）

### 2.1 获取组织树
- 请求：`GET /api/orgs/tree`
- Header：`Authorization: Bearer <token>`
- 检查点：
  - 返回 200，为树形结构（`children`）
  - 排序字段：`sortNo`

### 2.2 组织增删改
- 新增：`POST /api/orgs`
- 编辑：`PUT /api/orgs/{id}`
- 删除：`DELETE /api/orgs/{id}`
- Header：
  - `Authorization: Bearer <token>`
  - 删除额外需要：`X-Second-Verify: <secondVerifyToken>`
- 检查点：
  - 新增返回 201
  - 删除返回 204
  - 有子组织/有成员时删除应失败（提示信息明确）

### 2.3 组织成员视图（双向关联的交互验证）
- 查看成员：`GET /api/orgs/{orgId}/users`
- 添加成员（将用户归属组织改为 orgId）：`PUT /api/orgs/{orgId}/users/{userId}`
- 移除成员（将用户归属组织回退到父组织或根组织）：`DELETE /api/orgs/{orgId}/users/{userId}`
- Header：`Authorization: Bearer <token>`
- 检查点：
  - 成员列表与用户列表（按 orgId）一致

## 3. 用户管理（含导入导出、二次验证）

### 3.1 查询用户列表
- 请求：`GET /api/users?orgId=&keyword=`
- Header：`Authorization: Bearer <token>`
- 检查点：
  - 返回 200
  - 不应返回 `password` 字段
  - 数据权限生效（不同角色在同 orgId 上的可见范围符合配置）

### 3.2 新增/编辑用户（密码 BCrypt）
- 新增：`POST /api/users`（201）
- 编辑：`PUT /api/users/{id}`（200）
- Header：`Authorization: Bearer <token>`
- 检查点：
  - 新增时密码必填，后端落库为 BCrypt（数据库中应以 `$2a$/$2b$/$2y$` 开头）
  - 编辑时若未传密码，不应覆盖原密码
  - 角色配置：新增/编辑可带 `roleIds`（多角色）

### 3.3 导出用户（CSV）
- 请求：`GET /api/users/export?orgId=`
- Header：`Authorization: Bearer <token>`
- 检查点：
  - 返回 200，`Content-Type: text/csv`
  - CSV 表头：`id,username,realName,phone,email,avatar,orgId,status`

### 3.4 导入用户（CSV）
- 请求：`POST /api/users/import`
- Header：`Authorization: Bearer <token>`
- FormData：`file=<csv文件>`
- 检查点：
  - 返回 200，包含 `created`
  - 允许导入明文密码，后端落库会 encode

### 3.5 敏感操作二次验证（删除用户/删除组织）
- 获取二次验证 token：`POST /api/auth/second-verify`
  - Header：`Authorization: Bearer <token>`
  - Body：`{ "password": "..." }`
  - 返回：`{ "token": "..." }`
- 删除用户：`DELETE /api/users/{id}`
  - Header：`Authorization: Bearer <token>` + `X-Second-Verify: <secondVerifyToken>`
  - 返回：204
- 检查点：
  - 未传 `X-Second-Verify` 应失败（提示需要二次验证）
  - token 过期/错误应失败

## 4. 角色与权限（菜单/按钮、数据权限）

### 4.1 角色列表
- 请求：`GET /api/roles`
- Header：`Authorization: Bearer <token>`
- 检查点：返回 200，含 `name/code/status`

### 4.2 菜单树
- 请求：`GET /api/menus/tree`
- Header：`Authorization: Bearer <token>`
- 检查点：返回 200，含菜单与按钮节点（`type=MENU/BUTTON`）

### 4.3 角色绑定菜单/按钮权限
- 查询：`GET /api/roles/{roleId}/menus`
- 保存：`PUT /api/roles/{roleId}/menus`，Body：`{ "ids": [menuId...] }`
- Header：`Authorization: Bearer <token>`
- 检查点：
  - 保存后重新登录/刷新 `me`，前端菜单与按钮显隐变化一致

### 4.4 角色数据权限（按模块）
- 查询：`GET /api/roles/{roleId}/data-scopes`
- 保存：`PUT /api/roles/{roleId}/data-scopes`
- Header：`Authorization: Bearer <token>`
- 检查点：
  - 配置 `module_code=USER` 为 `ALL/ORG_AND_CHILDREN/ORG_ONLY/SELF_ONLY` 后，`GET /api/users` 的返回范围生效

## 5. 日志与审计
- 登录日志：`GET /api/logs/login`（返回最近 200）
- 操作日志：`GET /api/logs/operations`（返回最近 200）
- Header：`Authorization: Bearer <token>`
- 检查点：
  - 登录接口成功/失败均有记录
  - 用户/组织/角色/菜单等关键变更有操作日志记录（模块、动作、详情）
